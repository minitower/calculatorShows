Общая информация

  • Язык приложения: Python (серверная часть, модель прогнозирования, формирование конечных HTML файлов, тестирование и прочее), SQL (создание запросов к БД), CSS (создание таблицы стилей для конечных HTML файлов) и HTML (создание шаблонов веб-страниц);
  • Основные вспомогательные инструменты: Flask (серверная часть и формирование конечных HTML файлов), ClickHouse (организация БД, обработка запросов на получение данных), OpenVPN (организация маршрутов по сети для подключения к БД), Waitress (WSGI-сервер и обработка запросов к приложению Flask);
  • Общее описание модели:
    • Модель для прогнозирование количества просмотров: модель Хольта-Уинтерса (тройное экспоненциальное сглаживание);
    • Модель учета влияния CPA на количество просмотров: модифицированная модель сравнительного анализа;
    • Модель учета влияния апрува на вывод модели: модифицированная модель факторного анализа;
  • Допустимые HTTP-запросы: GET & POST для URL http://host:port/, для всех остальных - GET;
  • Хранение данных: ClickHouse (для получения статистики и ее первичное предобработки) и временные файлы;

Архитектура приложения

В общем случае можно сказать, что приложение содержит ML модель (собственно, прогнозирование количества показов для кампании и влияние факторов), серверную часть и UI выхода программы. Общение модулей программы происходит с помощью функций Python. Использование данного метода обусловлено тем, что приложение написано на чистом Python, что делает излишним создание RESTful API сервера.

Формирование UI происходит из HTML шаблонов, которые заполняются данными с помощью Flask. Для стилизации приложения используются CSS файлы, путь до который прокладывает Flask (для избегания проблемы несоответствия MIME-типов файлов);

Используемые библиотеки/фреймворки

Для работы различных модулей приложения необходимы следующие библиотеки (фреймворки):

  • Flask;
  • pandas: помогает парсить таблицы из ClickHouse, упрощает работу с данными и формирует HTML-таблицу c результатами;
  • sklearn: расчет ошибки прогноза,
  • scipy: формирование функции минимизации ошибки;
  • clickhouse-driver: настройка подключения к базе данных;
  • python-dotenv (dotenv): управление переменными окружения серверной части;
  • numpy: упрощение работы с временным рядом и таблицам pandas;
  • plotly: формирование графиков для временного ряда объема показов;
  • waitress: создание и обслуживание WSGI-сервера и обеспечение общения клиента и сервера;
  • requests: формирование запросов к серверу (для CI).

Стоит отметить, что в данном списке пропущены стандартные библиотеки Python. Узнать об их использовании можно в исходном коде программы, а также в комментариях и мануалах к различным классам и функциям.

Карта приложения

Статистические модели приложения

Как было описано выше, в данном приложении было использовано 3 модели, которые каким-либо образом влияют на конечные метрики, которые и являются выходом программы. При этом стоит отметить, что каждая из моделей влияет на свой уровень расчета. По отдельности данные модели не имеют практической значимости, так как не смогут в своей полноте дать верный результат для расчета. Исключением является модель прогнозирования количества просмотров: если CPA и аппрув не были получены от пользователя, то будет работать исключительно данная модель.

Рассчитываемые моделями метрики

К данным метрикам относятся:

  • Количество показов;
  • Количество кликов;
  • Количество лидов;
  • Количество подтвержденных лидов.

Природа данных метрик такова, что из 1 любой метрики представляется возможным вычислить все остальные метрики, если известны необходимые коэффициенты. Представим взаимосвязь данных метрик с помощью таблицы:

Метрика 1 ⇒ Метрика 2ПоказыКликиЛидыПодтвержденные лиды
ПоказыSS*CTRS*CTR*CRS*CTR*CR*approve
КликиS/CTRCC*CRC*CR*approve
ЛидыL/CTR/CRL/CRLL*CL
Подтвержденные лидыCL/CTR/CR/approveCL/CR/approveCL/approveCL

Соответственно, для показов определенной формулы не существует. Это объясняет самостоятельность модели прогнозирования показов и зависимость других моделей данного приложения.

Модель прогнозирования количества просмотров

Математическое описание модели и обоснование ее выбора

Данная модель несет центральное значение в приложении. В случае, если прогноз по данной модели будет ошибочным, то это отразится на всех метриках в выходе программы. Следовательно, точность данной модели является критично важной. В сравнении результатов анализа были использованы:

  • Линейная регрессия с параметрами:
    • shows, CPA (R2‾=85%\overline{R^2} = 85\%R2=85%﻿);
    • shows, CPA, EPC (R2‾=63%\overline{R^2} = 63\%R2=63%﻿);
    • shows, CPA, eCPM (R2‾=71%\overline{R^2} = 71\%R2=71%﻿);
    • shows, eCPM (R2‾=68%\overline{R^2} = 68\%R2=68%﻿).
  • Прогнозирование количества показов по предыдущим значением данной метрики (прогнозирование временного ряда):
    • AR (R2‾=91%\overline{R^2} = 91\%R2=91%﻿);
    • ARMA (R2‾=95%\overline{R^2} = 95\%R2=95%﻿);
    • Holt-Winters (R2‾=99.9%\overline{R^2} = 99.9\%R2=99.9%﻿).

Также были опробованы и другие модели, точность которых достаточно тяжело оценить с использованием R2‾\overline{R^2}R2﻿ : генетические алгоритмы, LRNN и другие.

Помимо временного ряда модель обязательно принимает на вход параметры:

  • Длина сезона (slen) - время, за которое, предположительно, временной ряд зациклится, то есть будет максимально равен предыдущему такому периоду. Данный период в модели фиксирован и составляет 24 часа;
  • Базисный вектор (x) - вектор параметров модели, которые будет подвержен обучению (обычно содержит предположения о временном ряде);
  • Длина временного ряда - количество точек во временном ряду.

Обучение модели Хольта-Уинтерса

Выше было сказано, что одним из входных параметров модели является базисный вектор. По своей сути данный вектор представляет из себя 3 числа, которые соответствуют параметрам α,β,γ\alpha, \beta, \gammaα,β,γ﻿ соответственно. Так как нам эти параметры неизвестны, то помимо конструирования данной модели необходимо также данную модель обучить. Для задачи обучения данной модели был выбран метод кросс-валидации, как современный, но в то же время надежный метод машинного обучения.

Помимо этого кросс-валидация эффективна тем, что ее оценка, т. е. средняя ошибка прогноза по нескольким проходам интерпретируема как вероятность ошибки в ее класическом понимании (т. е. удовлетворяет требованиям аксиоматики Колмагорова).

Об кросс-валидации можно говорить достаточно долго, что выходит за рамки документации. Здесь можно подробнее ознакомится с данным методом обучения и найти ответы на интересующие вопросы.
